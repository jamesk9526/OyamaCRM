## Purpose

DonorCRM today has several donor-engagement surfaces that overlap in confusing ways: Steward Paths, Communications, Letters & Printables, Email Builder, Tasks, Activities, donations, and dashboard attention widgets. They were built in waves and now share concepts (drafts, statuses, recipients, send queues, timeline events) without sharing contracts.

This document is the controlled refactor plan to make them feel like **one coordinated stewardship engine** without rewriting them all at once. It pairs with the new **Incremental Workspace Refactor Rules** in `AGENTS.md`.

The mental model the user-facing product should land on:

- **Steward Paths** — automation/orchestration brain
- **Communications** — email campaigns, blasts, drafts, send logs, email sequences
- **Letters & Printables** — printable materials, print queue, mail queue, form letters, PDFs, labels/envelopes
- **Tasks** — human work generated by paths or staff
- **Activities/Timeline** — source of truth for what actually happened

Steward Paths must **call** the others. It must not become a second email engine, a second print engine, or a second task list.

## Duplicated Concepts and Confusing User Paths

These are the overlaps the unified refactor must resolve:

1. **"Letters" tab inside Communications.** Communications today has a Letters tab. That made discovery easier short-term but blurs ownership. Letters & Printables is the source of truth; Communications should link out, not host letter management.
2. **Email drafts in two places.** Communications shows Email Drafts; Steward Paths has `StewardPathEmailDraft`. The user mental model should be: Steward Paths *creates* drafts that live in the Communications draft list and surface back in Steward Paths.
3. **Two engines.** `stewardPathsEngine.ts` (legacy) and `steward-paths-sequence-engine.ts` (sequences) both exist. Worker still calls both. Long-term we want one orchestration brain; short-term they coexist.
4. **"Send" semantics.** `SEND_EMAIL` step is implemented as draft-creation today. UI label and step name should match the safe behavior so users do not believe paths auto-send.
5. **Automations vs Steward Paths.** The Steward Paths page lives at `app/automations/page.tsx` and lists `Automation` rules. The naming and the URL diverge from the product language.
6. **Status vocabularies.** Backend enums (`StewardPathStatus`, `StewardPathStepRunStatus`, `StewardPathEmailDraftStatus`, generated-letter statuses, campaign statuses) are channel-specific. The shared user-facing status set is documented in `docs/DONOR_ENGAGEMENT_SYSTEM.md` but not consistently rendered yet.
7. **Visual builder is now present but partial on persistence.** The product now has a node-based workflow map (Palette / Canvas / Inspector) with branch lanes and condition editing. Branch persistence remains linear-only in this phase and is explicitly blocked from activation when unsupported.

## Workspace Ownership (target state)

### Letters & Printables OWNS
- Form letter templates, merge fields, header/footer/signature presets, branding
- Single + batch letter generation
- Generated letter review queue
- Print queue, mail queue
- PDF rendering/export, labels/envelopes
- Statuses: Draft, Needs Review, Approved, Generated, Queued For Print, Printed, Queued For Mail, Mailed, Failed, Archived

Letters & Printables MUST NOT host email blasts, email sequences, or marketing campaigns. The letter→email handoff stays as an explicit bridge action that creates a Communications draft.

### Communications OWNS
- Email campaigns, blasts, templates, drafts
- Review and approval, scheduled sends, recipient segments
- Send queue, delivery events, opens/clicks/bounces/unsubscribes
- Email sequence visibility when driven by Steward Paths

Communications MUST NOT manage physical print or mail queues. It may *show* that a path also generated a letter, with a link into Letters & Printables.

### Steward Paths ORCHESTRATES (does not duplicate)
Steward Paths owns the workflow builder, path templates, enrollments, step runs, timeline events, and automation logic. It calls existing tools rather than reimplementing them.

Target step/action types (existing, partial, and planned):

- Triggers: New donation, donor added to segment, donor becomes lapsed, pledge due, event attended, manual enrollment
- Timing: Wait N hours/days/weeks/months, wait until date, wait until weekday/time, wait until after last gift
- Email actions: create email draft, send review request, add to email sequence, schedule blast, wait for email open/click, mark email failed/bounced
- Print actions: generate form letter, add to print queue, require print approval, mark printed, add to mail queue, mark mailed
- Task actions: create task, assign staff member, wait for task completion, escalate overdue task
- Donor data actions: add tag, remove tag, update donor status, adjust engagement stage, add note
- Logic: if/else branch, donor segment, donation amount, communication preference, email engagement
- Safety/Review: require human approval, pause path, notify staff, stop enrollment, retry failed step

### Tasks OWNS
Manual follow-up work generated by paths or staff. Steward Paths creates tasks; tasks remain in the existing task system.

### Activities/Timeline OWNS
The source of truth for what actually happened. Every engagement surface writes Activity rows. Steward Paths timeline events are a secondary, path-scoped audit and should reference the canonical Activity record where one exists.

## Backwards Compatibility Requirements

The unified refactor must not break these contracts:

- Existing public routes:
  - `/communications` and `/communications/[campaignId]/*`
  - `/letters-printables/*`
  - `/automations` (current Steward Paths home)
- Existing API contracts:
  - `/api/email-campaigns/*`
  - `/api/letters/*` (templates, generated, queues, batches, create-email-draft)
  - `/api/steward-paths/*` (templates, steps, enrollments, timeline, drafts, process-due)
  - `/api/automations/*`
  - `/api/tasks/*`
- Persistence:
  - `Automation`, `StewardPath`, `StewardPathStep`, `StewardPathEnrollment`, `StewardPathStepRun`, `StewardPathTimelineEvent`, `StewardPathEmailDraft`, `EmailCampaign`, `GeneratedLetter`, `Activity`, `Task` — no destructive migrations in a single pass.
- Safety defaults:
  - Draft-first email behavior on `SEND_EMAIL` step
  - `doNotEmail` / `emailOptOut` recipient suppression in send paths
  - Audit + Activity write-backs on key transitions

When `/automations` is renamed to `/steward-paths`, a redirect or compatibility page must keep `/automations` working. When the "Letters" tab is removed from Communications, the Communications page must link to `/letters-printables` so discovery does not regress.

## Known Risks

- **Worker drift**: `steward-paths-worker.ts` still drives both the legacy engine and the sequence engine. Removing the legacy engine prematurely would break in-flight enrollments. Phase 3 must confirm parity before Phase 5 retires legacy paths.
- **Migration safety on MySQL with case-sensitive table names**: see `mysql-prisma-casing-rules`. Any new step-type enum value or column must use exact Prisma table casing.
- **Auto-send temptation**: adding `SEND_EMAIL` real send behavior must keep `email_auto_send` permission gate, organization setting, draft-first default, and recipient suppression. Phase 5 should not flip the default behavior.
- **UI confusion during transition**: removing the Letters tab from Communications without a clear pointer to `/letters-printables` would degrade discovery. Phase 2 must add the link card at the same time as the removal.
- **Visual-builder scope creep**: map canvas and branch lanes are now in place, but drag/drop polish, deeper history/settings tabs, and full branch persistence are explicit later passes.
- **Status label drift**: every workspace must use the shared status vocabulary. Adding new ad-hoc status strings in any new step UI would re-introduce divergence.
